Cadeia de Identidade — Workload Identity

Zero secrets. Zero connection strings no código. Autenticação via OIDC + Azure AD.

  1. Terraform cria a Managed Identity e a Federated Credential vinculada ao ServiceAccount do K8s
  2. O mutating webhook do Workload Identity injeta variáveis e o token OIDC no Pod
  3. DefaultAzureCredential no Java detecta automaticamente o token e faz a troca OIDC → Access Token
  4. O Access Token é usado nas chamadas ao Azure Storage com as roles RBAC atribuídas

Java/Quarkus — Acesso ao Blob Storage

Construção do Client

@ApplicationScoped
public class BlobStorageService {

  private final BlobContainerClient containerClient;

  public BlobStorageService(
      @ConfigProperty(name="upload.storage.blob-service-url")
      String blobServiceUrl,
      @ConfigProperty(name="upload.storage.uploads-container-name")
      String containerName) {

    this.containerClient = new BlobServiceClientBuilder()
        .endpoint(blobServiceUrl)
        .credential(new DefaultAzureCredentialBuilder().build())
        .buildClient()
        .getBlobContainerClient(containerName);
  }
}

Upload de CSV

public void upload(String blobPath,
                   InputStream data,
                   long length,
                   String contentType) {

  containerClient
    .getBlobClient(blobPath)
    .upload(data, length, false);

  containerClient
    .getBlobClient(blobPath)
    .setHttpHeaders(
      new BlobHttpHeaders()
        .setContentType(contentType));
}

Download (Worker)

// Receber
workQueue.receiveMessages(
    batchSize,
    Duration.ofSeconds(3600), // visibility
    null, null)
  .stream()
  .map(msg -> {
    String decoded = new String(
      Base64.getDecoder()
        .decode(msg.getBody().toString()));
    return objectMapper.readValue(
      decoded, UploadJobMessage.class);
  });

// Deletar após processamento
workQueue.deleteMessage(
    msg.getMessageId(),
    msg.getPopReceipt());

Role necessária: Storage Queue Data Contributor — permite send, receive, peek e delete. Mensagens são codificadas em Base64 (requisito do Azure Queue).

Autoscaling com KEDA

flowchart TB
    Q["📬 Queue\nwork-items"] -->|"queueLength = 1"| KEDA["⚡ KEDA\nScaledObject"]
    KEDA -->|"HPA"| DEP["☸ Deployment\nworker"]
    DEP -->|"0 → 10 pods"| PODS["🔧🔧🔧\n...N pods"]

Configuração

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
spec:
  scaleTargetRef:
    name: worker
  pollingInterval: 15
  cooldownPeriod: 300
  minReplicaCount: 0
  maxReplicaCount: 10
  triggers:
    - type: azure-queue
      metadata:
        queueName: work-items
        queueLength: "1"

Comportamento Observado

  1. Fila vazia: 0 workers (scale to zero)
  2. 10 mensagens: KEDA escala para 10 workers
  3. Cluster Autoscaler adiciona nós se não houver CPU/RAM
  4. Cooldown 5min: após fila vazia, workers são removidos
